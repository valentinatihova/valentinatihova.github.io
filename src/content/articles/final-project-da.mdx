# Mobile App User Behavior & A/B Testing

# Analysis of user behavior in the mobile application "Unnecessary things"

The dataset contains data about events (mobile_dataset) committed in the “Unnecessary Things” mobile application. In it, users sell their unwanted items by posting them on a bulletin board. In this case, the mobile application can be installed from different sources (mobile_sources).

The dataset contains data from users who first performed actions in the application after October 7, 2019.

### Goals:
- Analyze the relationship between the target event - viewing contacts - and other user actions.
- The customer of this research wants to see what the portrait of clients who are part of the “hot” churn (they actively used the application and left the application recently) is;
- Based on our analysis, targeted programs will be applied to those application clients who do not view contacts and churn (or are close to it) to retain clients and increase their involvement.

### Tasks:
1. Conduct exploratory data analysis (EDA stage).
2. Based on customer behavior, determine what actions lead to viewing the seller’s contacts.
3. Test statistical hypotheses.
    
### Project stages:
1. [Data preprocessing](#Data_preprocessing):
    - investigate missing values;
    - see data types;
    - look for duplicates in the data (and process them if necessary);
    - correct the names of the columns (to increase readability);
    - look for gaps in the data.
2. [Exploratory EDA Data Analysis](#EDA):
    - see what actions users perform in the application, with what frequency, their [correlation](#corr) at the user level;
    - look at the clients that are represented in the datasets: what groups users can be divided into depending on the completion of the target event (viewing a contact);
    - you can divide users into groups by looking at the common characteristics of clients and their correlation, or you can use [clustering](#ML) ML tools;
    - we build a funnel for clients, look at the relationship between actions;

3. [Working with metrics and business indicators](#metrics): for each group, look at the engagement and activity metrics and group validation (if the differences between groups are small, it may be worth considering an alternative breakdown):
    - customer return retention rate;
    - duration of the user session in the application - look at the time interval between the start of sessions and see the distribution among clusters (if the interval is more than N hours (minutes), then we consider that the client closed the application);
    - frequency of actions, 
    - conversion.
4. [Testing hypotheses](#hyp): with those users who have churn (or are close to it), we begin to test hypotheses based on user actions.
    - [hypothesis 1](#hyp1): conversion to contact views differs between these two groups (Some users perform the actions tips_show and tips_click, others - only tips_show);
    - [hypothesis 2](#hyp2): according to sources of attracting users - the problem of self-selection - users of different sources may have a different sequence of events.    
5. Obtaining a ["portrait" of a client](#summary) who is churn and formulate actions to retain him.

To select 'sessions' from the event log and calculate its duration, it is necessary.
    
1) To do this, sort them by user_id, then the events of each user will go sequentially.

2) In the 'session_duration' column, for each event of an individual user, we calculate the difference between the time of visiting the page and the time of visiting the previous page. If the page was the first for the user, then the value in the 'session_duration' column will be NaT, since there is no previous value.

3) We find events that will be the starting points for sessions (NaT).

4) From the main dataframe 'df' we will create an auxiliary dataframe 'session_start_df'. This dataframe will contain events that will be considered the first events of the sessions. Such events include all events that occurred more than `30` minutes after the previous one, or events that were the first for the user (NaT in the ’session_duration’ column). We will also create a column 'session_id' in the auxiliary dataframe, which will contain the id of the first event of the session. It will be useful to correctly display the session identifier when we combine data from the main and auxiliary dataframes.

5) Using the merge_asof function, we will combine the data of the main and auxiliary dataframes. This function allows you to combine data from two dataframes in a similar way to a left join, but not by exact match of keys, but by the closest one.

Let's put all the actions into the `session_duration()` function.

```python
def session_duration(df):
    
    """ Функция расчета длительности сессии  """
    
    # 1
    df = df.sort_values(['user_id', 'event_time'])
    df['id'] = df.index

    # 2
    df['session_duration'] = df.groupby('user_id')['event_time'].diff(1)

    # 3-4
    sessions_start_df = df[(df['session_duration'].isnull()) | (df['session_duration'] > '1800 seconds')]
    sessions_start_df['session_id'] = sessions_start_df.index
    sessions_start_df['id'] = sessions_start_df.index

    # 5
    df = df.sort_values('id')
    sessions_start_df = sessions_start_df.sort_values('id')
    df = pd.merge_asof(df, sessions_start_df[['id','user_id','session_id']], on='id', by='user_id')

    df['session_duration_min'] = df['session_duration'].astype('timedelta64[m]') # переводим время в минуты
    
    plt.figure(figsize=(16,5));
    sns.boxplot(df.groupby('session_id')['session_duration_min'].sum());
    plt.title('Средняя длительность сессии, минуты', loc='left', pad=12, fontweight="bold");
    sns.set()
    return plt.show()
```

Let's write a function that will count and show customer retention by day `RetentionaRate()`.

```python
def RetentionaRate(df):
    
    """ Функция расчета показателя удержания клиента:
        пользователей выделяют в когорты по дате первого
        его действия, определяют кол-во новых 
        пользователей по дням                             
    """   
    
    df['event_time'] = pd.to_datetime(df['event_time'])
    first_date = df.groupby(['user_id'])['event_time'].min()
    first_date.name = 'first_visit_date'
    df = df.join(first_date, on='user_id')

    df['visit_day'] = df['event_time'].astype('datetime64[D]')
    df['first_visit_day'] = df['first_visit_date'].astype('datetime64[D]')

#      lifetime пользователя в рамках когорты
    df['cohort_lifetime'] = df['visit_day'] - df['first_visit_day']
    df['cohort_lifetime'] = df['cohort_lifetime'] / np.timedelta64(1,'D')
    df['cohort_lifetime'] = df['cohort_lifetime'].round().astype('int')
    
#     для каждой когорты количество активных пользователей
    cohorts = df.groupby(['first_visit_day','cohort_lifetime']).agg({'user_id':'nunique'}).reset_index()

#     исходное количество пользователей в когорте
    initial_users_count = cohorts[cohorts['cohort_lifetime'] == 0][['first_visit_day','user_id']]
    initial_users_count = initial_users_count.rename(columns={'user_id':'cohort_users'})
    
#     Объединим данные по когортам с исходным количеством пользователей в когорте:
    cohorts = cohorts.merge(initial_users_count,on='first_visit_day')
    
#     рассчитаем Retention Rate
    cohorts['retention'] = cohorts['user_id']/cohorts['cohort_users']
    cohorts['first_visit_day'] = cohorts['first_visit_day'].dt.strftime('%Y-%m-%d')
    
    retention_pivot = cohorts.pivot_table(index='first_visit_day',
                                      columns='cohort_lifetime',
                                      values='retention',
                                      aggfunc='sum')
#     рисуем тепловую карту
    plt.figure(figsize=(7,5))
    plt.title('Как часто люди возвращаются?', loc='left', pad=12, fontweight="bold")
    sns.heatmap(retention_pivot,
                    annot=False, 
                    fmt='.1%', 
                    linewidths=1, 
                    linecolor='gray');
```

## Testing hypotheses <a name="hyp"></a>

Let's test several hypotheses regarding the conversion to views to contacts. To do this, we will write a function (part of the EDA() function) that will show the funnel of events and conversions along it, as well as a function that will check the statistical significance of the shares.

Let's separate the part of the `EDA()` function for calculating conversion into a separate `funnel()`.

```python
def funnel(df):
    
    """ Функция расчета конверсии в воронке событий  """
    
    events_per_user = (df.groupby('event_name')
                         .agg({'user_id': 'nunique'})
                         .reset_index().sort_values(by='user_id', ascending = False)
                         .rename({'event_name': 'event', 'user_id': 'count_of_users'}, axis=1))

    events_per_user['unique_prev_step'] = events_per_user['count_of_users'].shift(1)

    events_per_user.loc[events_per_user['unique_prev_step'].isna(),'unique_prev_step'] = \
              events_per_user.loc[events_per_user['unique_prev_step'].isna(),'count_of_users']

    events_per_user['prev_step_%'] = round(events_per_user['count_of_users'] / events_per_user['unique_prev_step'] *100, 2)
    
    hist(events_per_user[['event', 'prev_step_%','count_of_users']].sort_values('count_of_users', ascending=True),
     'Конверсия по шагам воронки, в %', 'доля пользователей, в %', 'событие')
```

Let's look at the statistical significance of the shares using the z-test for each step of the funnel. For convenience, let's write a function.

```python
def z_test(successes1, successes2, trials1, trials2, alpha=0.01):
    
    """ Функция расчета статистической значимости
        долей каждого шага воронки z-критерием
    """
    
    # пропорция успехов в первой и второй группах:
    p1 = successes1/trials1
    p2 = successes2/trials2

    # пропорция успехов в комбинированном датасете:
    p_combined = (successes1 + successes2) / (trials1 + trials2)

    # разница пропорций в датасетах
    difference = p1 - p2
    
    # считаем статистику в ст.отклонениях стандартного нормального распределения
    z_value = difference / mth.sqrt(p_combined * (1 - p_combined) * (1/trials1 + 1/trials2))

    # задаем стандартное нормальное распределение (среднее 0, ст.отклонение 1)
    distr = st.norm(0, 1) 

    # считаем статистику в ст.отклонениях стандартного нормального распределения
    z_value = difference / mth.sqrt(p_combined * (1 - p_combined) * (1/trials1 + 1/trials2))

    p_value = (1 - distr.cdf(abs(z_value))) * 2

    print(f'p-значение: {p_value:.3%}')

    if (p_value < alpha):
        print('Отвергаем нулевую гипотезу: между долями есть значимая разница')
    else:
        print('Не получилось отвергнуть нулевую гипотезу, нет оснований считать доли разными')
```

def check_hypithesis(df, event_name, exp1, exp2):
    
    """ Функция проверки гипотезы для двух
        экспериментальных групп
    """    
    
    users_events_per_group = df.pivot_table(index='event_name', 
                                        values='user_id', 
                                        columns='group', 
                                        aggfunc=lambda x: x.nunique())

    gr_count_user = (df.pivot_table(index='group', values='user_id', aggfunc=lambda x: x.nunique()))['user_id']
    
    pr1 = users_events_per_group.loc[event_name, exp1] / gr_count_user.loc[exp1]
    pr2 = users_events_per_group.loc[event_name, exp2] / gr_count_user.loc[exp2]
    
    print(f'{pr1:.0%} пользователей с событием {event_name} в группе {exp1}')
    print(f'{pr2:.0%} пользователей с событием {event_name} в группе {exp2}')
    
    z_test(users_events_per_group.loc[event_name, exp1],
           users_events_per_group.loc[event_name, exp2],
           gr_count_user.loc[exp1],
           gr_count_user.loc[exp2])

## General conclusions <a name="summary"></a>

In total, during the study it became clear that the fact of viewing the seller’s contacts is most influenced by such events as: the ad map (the client navigates the map much faster and makes decisions about contacting the seller), viewing the ad card and their photos, and, of course, recommendations for the ad (viewing them and opening them).


In the process of grouping users into groups based on characteristic characteristics, it was possible to find out that customer activity drops towards the end of the month, with a sharp decline observed in November. The most “active” users look at photos of ads, view and click on recommended ads, and also perform search actions 1 (using various hashtags?). 
Such users often use the application and update/open it repeatedly over time (especially if they have added the ad to their favorites).


For users who are churn or close to it, they probably can’t figure out the search actions for ads, use different types of searches, it bores them, and they simply don’t want to continue using the application. Yes, they view recommended ads, but don't get to the point of viewing photos. This is due to the fact that to view photos you must go to the ad card. This step in the funnel is clearly unnecessary and can be excluded.


There is a possibility that the source of attraction is reflected in user actions. The actions of users who downloaded the application using Yandex and Google have similar actions in the event funnel and conversion values ​​to the contact viewing step. Users from other sources clearly differ in how they use the app. With other sources, you need to work out the steps of the funnel so that their actions are clear and logical for application clients.
